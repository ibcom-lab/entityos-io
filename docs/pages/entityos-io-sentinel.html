entityOS · Sentinel

The IP Set firewall that maintains itself.

Sentinel treats the public internet as hostile and admits only registered, verified origins. Five functions form one closed loop — issuing access codes, registering IPs, processing them into rules, compiling the allowlist, and enforcing it at the edge.

One closed loop

Access is earned, then continuously renewed.

Two layers keep the allowlist live — the AWS IP Set update that compiles and enforces it, and the self-management pipeline that lets access be requested, verified and processed into rules.

The five functions

Five functions. One firewall.

Each function is a discrete entityOS Sentinel method with its own inputs, cadence and output. Together they keep a single AWS IP Set current.

1
sentinel-firewall-ip
Compile the allowlist

The writer of the firewall. It gathers three sources of truth — recent Requests History (R), each user's Last IP (U), and the explicit Network Access Rules (N) — and reconciles them into a single AWS IP Set. The set is produced in both scopes: Regional for the ALB and Global for CloudFront, so the same allowlist governs every edge.

InputsRequests History · Users Last IP · Network Access Rule
CadenceUPDATE 15 min
OutputAWS IP Set — Regional (ALB) + Global (CloudFront)
2
sentinel-access
Enforce at the edge

Where the IP Set becomes a wall. An AWS WAF Rule references the compiled set and is attached to the edge so any origin not on the allowlist is dropped before it reaches an application. Two deployment topologies share the same rule: the global path (ip.access.entityos.io → CloudFront → Lambda) is live, and the regional path (*.entityos.cloud → ALB → EC2) extends the same control to server workloads.

Gates onAWS WAF Rule → AWS IP Set
Globalip.access.entityos.io → CloudFront → Lambda
Regional*.entityos.cloud → ALB → EC2
3
sentinel-conductor-security-codes
Issue and store access codes

The credential factory. For each Space and User it mints a GUID, then splits it two ways: the raw GUID is sent to the Space admin as a security code for out-of-band distribution, while only its SHA-256 hash is written to the Security Hashes List in S3. The plaintext code is never persisted by the platform — the stored hash is what a registration is later checked against.

InputsSpace · User → GUID
CadenceUPDATE 60 min
Processsentinel-conductor-get-security-codes-from-entityos
OutputsSHA256 → S3 Security Hashes List · GUID → Space admin
4
sentinel-register
Register an IP with a code

The public intake endpoint at ip.register.entityos.io. A user submits their current IP together with the security code they were given. The submitted code is hashed and checked against the Security Hashes List from sentinel-conductor-security-codes; only if the hash matches is the request accepted and persisted as a JSON IP Registration in S3 — the record the conductor then turns into a Network Access Rule.

Endpointip.register.entityos.io
ValidatesSubmitted code → SHA256 → S3 Security Hashes List
OutputS3 JSON IP Registration
5
sentinel-conductor-network-access
Process registrations into rules

The conductor that promotes a raw registration into an active permission. It reads the JSON IP registrations produced by sentinel-register and, scoped to the correct Space and User, writes the corresponding Network Access Rule. Those rules are exactly the (N) source that sentinel-firewall-ip compiles — closing the path from a public submission to an enforced allowlist entry.

InputsSpace · User · JSON IP Registration
CadenceUPDATE 60 min
Processsentinel-conductor-process-registrations
OutputNetwork Access Rule

The takeaway

A firewall that stays current on its own.

No hand-edited lists, no stale rules. Access is issued, verified, and continuously reconciled into the edge.

Sentinel turns network access into a self-maintaining loop: a code is issued, an IP is registered against its hash, the registration becomes a rule, the rules compile into the IP Set every 15 minutes, and the WAF enforces it. Only registered, verified origins ever reach an application.
